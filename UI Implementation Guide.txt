1.	System logic (reference)
•	Boundaries
•	Central service (AuthDbContext): users/roles/permissions, tenants, tenant subscriptions, tenant nodes, billing webhooks, central authoritative tenant & billing state.
•	Local hospital service (HmsDbContext): patients, visits, invoices, prescriptions, local users/roles/permissions cache, local subscription cache, local operations.
•	Authentication
•	Human login normally goes to central via AuthService and issues central JWT with tenant_id claim.
•	Local login uses LocalAuthService -> issues a local JWT signed with LocalJwt:Key. Local tokens include roles/permission claims from local cache.
•	Authorization
•	Permission-based attribute [HasPermission("...")] backed by JWT permission claims and PermissionAuthorizationHandler.
•	Local authorization works offline because local JWT contains cached roles/permissions.
•	Tenancy
•	TenantId is the single source of truth. CurrentTenantAccessor ambient context set by middleware (from JWT or X-Tenant-Id header).
•	EF global query filters enforce tenant scoping (allowing TenantId == null for central/shared data).
•	Subscriptions
•	TenantSubscription central record; middleware checks ITenantSubscriptionService.IsTenantAllowedAsync and blocks requests with HTTP 402 if inactive.
•	Seed creates sample subscription records; webhook endpoint /subscriptions/webhook accepts signed events to update central subscription state.
•	Sync & Outbox
•	Local background sync: BackgroundSyncService runs SyncManager.RunOnceAsync() periodically, calls ICloudSyncClient (push local unsynced records, pull remote changes).
•	Outbox: local changes that need distribution are saved to OutboxMessage. OutboxProcessor publishes to event publisher and can push to tenant nodes.
•	Push notifier: central attempts to push events to registered tenant nodes (callback URLs) using TenantNode.CallbackSecret HMAC signature (X-Central-Signature).
•	Manual sync: POST /sync/tenant/{tenantId}/sync-now triggers tenant-scoped sync; ISyncManager.RunOnceAsync(Guid tenantId) implemented to pull subscription/user profile and local user sync.
•	Local user management & offline
•	LocalUser and LocalRole/LocalPermission exist in local DB for offline login and offline role/permission enforcement.
•	Local user changes are pushed to central during sync; central can push subscription/other changes to nodes via outbox/push or nodes poll central.
•	Security
•	JWTs: central tokens signed with Jwt:Key; local tokens signed with LocalJwt:Key. Keep keys secure and rotate.
•	Webhook/push HMAC: central signs webhook pushes with node CallbackSecret. Nodes must verify signature.
•	Protect all management endpoints with users.manage or admin scopes.
•	Operational notes
•	Two DBs per deployment: central auth DB; local Hms DB per hospital. Migrations must be applied to the correct context.
•	Sync uses eventual consistency; outbox + push improves timeliness when reachable.

•	Use a responsive front-end (Bootstrap 5 is good and works with Razor Pages). Use CSS variables for theme switching (dark/light).
•	Simple header with: hospital name, tenant badge, sync status indicator, user menu (profile/logout).

4.	Mapping UI → APIs (examples)
•	Login page:
•	POST /auth/login (central) when online. If fail due to network, fallback to POST /localauth/login (local).
•	On local login receive local JWT, store in cookie, set CurrentTenant, and load local roles from /localroles or token claims.
•	Users page:
•	GET /localusers?tenantId={tenantId} list
•	POST /localusers create
•	PUT /localusers/{id}/lock etc.
•	Sync page:
•	GET /sync/status (admin) for last run
•	POST /sync/tenant/{tenantId}/sync-now to trigger immediate sync
•	Tenants:
•	GET/POST /tenants; token issuance /tenants/{id}/token
•	Subscriptions:
•	GET/POST/PUT /subscriptions/{tenantId} and webhook /subscriptions/webhook
5.	Offline handling patterns for UI
•	Detect network status via JS: navigator.onLine + periodic ping to /health or a small endpoint.
•	Cache last-known state / timestamps in localStorage (or IndexedDB) for UI.
•	Retry queue: for local admin operations create a small client-side queue that retries when online — server already synchronizes via background sync, but UI-level retry improves UX.
•	Token selection: prefer central JWT; if offline and local JWT exists and valid, use local.
6.	Security & storage
•	Keep JWTs in Secure, HttpOnly cookies (not localStorage) unless SPAs; Razor Pages can set cookies easily.
•	Store node secrets only on operator machines; central shows secret once upon registration.
•	Use TLS everywhere.
